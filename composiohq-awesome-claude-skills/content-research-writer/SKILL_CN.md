# 内容研究写作者

此技能作为您的写作伙伴，帮助您研究、概述、起草和完善内容，同时保持您独特的声音和风格。

## 何时使用此技能

- 撰写博客文章、文章或通讯
- 创建教育内容或教程
- 起草思想领导力文章
- 研究和撰写案例研究
- 制作带有来源的技术文档
- 撰写带有适当引用和参考的内容
- 改进钩子和介绍
- 在写作时获得逐节反馈

## 此技能的功能

1. **协作大纲**：帮助您将想法结构化为连贯的大纲
2. **研究协助**：查找相关信息并添加引用
3. **钩子改进**：加强您的开头以吸引注意力
4. **章节反馈**：在您写作时审查每个部分
5. **声音保留**：保持您的写作风格和语调
6. **引用管理**：正确添加和格式化参考
7. **迭代改进**：帮助您通过多个草稿进行改进

## 如何使用

### 设置您的写作环境

为您的文章创建一个专用文件夹：
```
mkdir ~/writing/my-article-title
cd ~/writing/my-article-title
```

创建您的草稿文件：
```
touch article-draft.md
```

从这个目录打开 Claude Code 并开始写作。

### 基本工作流程

1. **从大纲开始**：
```
Help me create an outline for an article about [topic]
```

2. **研究并添加引用**：
```
Research [specific topic] and add citations to my outline
```

3. **改进钩子**：
```
Here's my introduction. Help me make the hook more compelling.
```

4. **获取章节反馈**：
```
I just finished the "Why This Matters" section. Review it and give feedback.
```

5. **完善和润色**：
```
Review the full draft for flow, clarity, and consistency.
```

## 说明

当用户请求写作帮助时：

1. **理解写作项目**
   
   询问澄清问题：
   - 主题和主要论点是什么？
   - 目标受众是谁？
   - 期望的长度/格式是什么？
   - 你的目标是什么？（教育、说服、娱乐、解释）
   - 有任何现有的研究或来源需要包含吗？
   - 你的写作风格是什么？（正式、对话式、技术性）

2. **协作大纲**
   
   帮助构建内容：
   
   ```markdown
   # 文章大纲：[标题]
   
   ## 钩子
   - [开场行/故事/统计数据]
   - [读者为什么应该关心]
   
   ## 介绍
   - 背景和上下文
   - 问题陈述
   - 本文涵盖内容
   
   ## 主要部分
   
   ### 第 1 部分：[标题]
   - 关键点 A
   - 关键点 B
   - 例子/证据
   - [需要研究：特定主题]
   
   ### 第 2 部分：[标题]
   - 关键点 C
   - 关键点 D
   - 需要数据/引用
   
   ### 第 3 部分：[标题]
   - 关键点 E
   - 反论点
   - 解决方案
   
   ## 结论
   - 主要点摘要
   - 行动号召
   - 最终思考
   
   ## 研究待办事项
   - [ ] 查找 [主题] 的数据
   - [ ] 获取 [概念] 的例子
   - [ ] 为 [声明] 寻找引用
   ```
   
   **迭代大纲**：
   - 根据反馈调整
   - 确保逻辑流程
   - 识别研究差距
   - 标记需要深入研究的部分

3. **进行研究**
   
   当用户请求关于某个主题的研究时：
   
   - 搜索相关信息
   - 找到可靠的来源
   - 提取关键事实、引述和数据
   - 以请求的格式添加引用
   
   示例输出：
   ```markdown
   ## 研究：AI 对生产力的影响
   
   关键发现：
   
   1. **生产力提升**：研究显示内容创建任务节省 40% 的时间 [1]
   
   2. **采用率**：67% 的知识工作者每周使用 AI 工具 [2]
   
   3. **专家引述**："AI 增强而非取代人类创造力" - 麻省理工学院 Jane Smith 博士 [3]
   
   引用：
   [1] 麦肯锡全球研究所。(2024)。"生成式 AI 的经济潜力"
   [2] Stack Overflow 开发者调查 (2024)
   [3] Smith, J. (2024)。麻省理工科技评论采访
   
   添加到大纲的第 2 部分。
   ```

4. **改进钩子**
   
   当用户分享介绍时，分析并加强：
   
   **当前钩子分析**：
   - 什么有效：[积极元素]
   - 什么可以更强：[改进领域]
   - 情感影响：[当前 vs. 潜力]
   
   **建议的替代方案**：
   
   选项 1：[大胆声明]
   > [示例]
   *为什么有效：[解释]*
   
   选项 2：[个人故事]
   > [示例]
   *为什么有效：[解释]*
   
   选项 3：[令人惊讶的数据]
   > [示例]
   *为什么有效：[解释]*
   
   **钩子问题**：
   - 它创造好奇心吗？
   - 它承诺价值吗？
   - 它足够具体吗？
   - 它与受众匹配吗？

5. **提供逐节反馈**
   
   当用户撰写每个部分时，审查：
   
   ```markdown
   # 反馈：[部分名称]
   
   ## 什么工作得很好 ✓
   - [优势 1]
   - [优势 2]
   - [优势 3]
   
   ## 改进建议
   
   ### 清晰度
   - [具体问题] → [建议的修复]
   - [复杂句子] → [更简单的替代方案]
   
   ### 流程
   - [过渡问题] → [更好的连接]
   - [段落顺序] → [建议的重新排序]
   
   ### 证据
   - [需要支持的声明] → [添加引用或示例]
   - [通用声明] → [使其更具体]
   
   ### 风格
   - [语调不一致] → [更好地匹配你的声音]
   - [词汇选择] → [更强的替代方案]
   
   ## 具体行编辑
   
   原文：
   > [草稿中的准确引述]
   
   建议：
   > [改进版本]
   
   原因：[解释]
   
   ## 思考问题
   - [发人深省的问题 1]
   - [发人深省的问题 2]
   
   准备好进入下一部分！
   ```

6. **保留作者的声音**
   
   重要原则：
   
   - **学习他们的风格**：阅读现有的写作样本
   - **建议，不要替换**：提供选项，而不是指令
   - **匹配语调**：正式、随意、技术性、友好
   - **尊重选择**：如果他们更喜欢自己的版本，支持它
   - **增强，不要覆盖**：让他们的写作更好，而不是不同
   
   定期询问：
   - "这听起来像你吗？"
   - "这是正确的语调吗？"
   - "我应该更/更少 [正式/随意/技术性] 吗？"

7. **引用管理**
   
   根据用户偏好处理引用：
   
   **内联引用**：
   ```markdown
   研究显示 40% 的生产力提升（麦肯锡，2024）。
   ```
   
   **编号引用**：
   ```markdown
   研究显示 40% 的生产力提升 [1]。
   
   [1] 麦肯锡全球研究所。(2024)...
   ```
   
   **脚注风格**：
   ```markdown
   研究显示 40% 的生产力提升^1
   
   ^1: 麦肯锡全球研究所。(2024)...
   ```
   
   维护运行中的引用列表：
   ```markdown
   ## 参考
   
   1. 作者。(年份)。"标题"。出版物。
   2. 作者。(年份)。"标题"。出版物。
   ...
   ```

8. **最终审查和润色**
   
   当草稿完成时，提供全面的反馈：
   
   ```markdown
   # 完整草稿审查
   
   ## 总体评估
   
   **优势**：
   - [主要优势 1]
   - [主要优势 2]
   - [主要优势 3]
   
   **影响**：[整体有效性评估]
   
   ## 结构与流程
   - [组织评论]
   - [过渡质量]
   - [节奏评估]
   
   ## 内容质量
   - [论点强度]
   - [证据充分性]
   - [示例有效性]
   
   ## 技术质量
   - 语法和 mechanics：[评估]
   - 一致性：[评估]
   - 引用：[完整性检查]
   
   ## 可读性
   - 清晰度评分：[评估]
   - 句子多样性：[评估]
   - 段落长度：[评估]
   
   ## 最终润色建议
   
   1. **介绍**：[具体改进]
   2. **正文**：[具体改进]
   3. **结论**：[具体改进]
   4. **标题**：[必要时的选项]
   
   ## 发布前检查清单
   - [ ] 所有声明都有来源
   - [ ] 引用已格式化
   - [ ] 示例清晰
   - [ ] 过渡流畅
   - [ ] 包含行动号召
   - [ ] 校对拼写错误
   
   准备发布！🚀
   ```

## 示例

### 示例 1：Teresa Torres 的工作流程

**用户**："我正在写一篇关于持续发现的文章。帮我创建一个大纲。"

**流程**：
1. 协作制定大纲结构
2. 识别研究需求
3. 用户开始撰写介绍
4. 审查并改进钩子
5. 用户撰写每个部分
6. 每个部分后提供反馈
7. 进行研究并添加引用
8. 完整草稿的最终审查
9. 润色并准备发布

**结果**：研究充分、引用适当的文章，以 Teresa 的声音写成，结构和流程强大。

### 示例 2：研究密集型文章

**用户**："我正在写关于 AI 对产品管理的影响。帮我找到当前的数据和例子。"

**输出**：
```markdown
## 研究汇编

### 最近数据
- Gartner：到 2025 年，80% 的 PM 将使用 AI 工具 [1]
- 调查：45% 报告更快的功能交付 [2]
- 案例研究：Airbnb 的 AI 辅助优先级排序 [3]

### 专家引述
- "AI 放大 PM 判断，而非取代它" - Marty Cagan
- [带有引用的其他引述]

### 真实例子
1. **公司 A**：使用 AI 进行用户研究合成
   - 结果：60% 时间节省
   - 来源：[引用]

2. **公司 B**：AI 驱动的路线图分析
   - 结果：更好的利益相关者对齐
   - 来源：[引用]

所有内容都已添加到您的大纲中，并带有适当的引用。
```

### 示例 3：钩子改进

**用户的原始钩子**：
> "产品管理因 AI 而改变。在本文中，我将讨论 AI 影响产品经理的一些方式。"

**改进选项**：

**选项 1（数据驱动）**：
> "上个月，我让 AI 分析 500 次客户访谈。它花了 30 分钟而不是 3 周。产品管理将永远不一样。"

**选项 2（问题）**：
> "如果你能与每个客户交谈，阅读每个评论，分析每个支持工单 - 都在你的早咖啡之前，会怎样？"

**选项 3（故事）**：
> "Sarah 花了两周时间构建了错误的功能。不是因为她不了解她的用户，而是因为她无法足够快地处理数百次访谈以发现模式。"

### 示例 4：章节反馈

**用户**："刚完成我的'常见错误'部分。想法？"

**响应**：
```markdown
# 反馈：常见错误部分

## 什么工作得很好 ✓
- 强有力的例子（Slack 通知故事）
- 清晰的结构（3 个错误，干净的布局）
- 每个错误的实用建议

## 建议

### 使其更具体
你的第二点说"糟糕的优先级"，但可以更强：

当前：
> "许多团队通过专注于功能而非结果来糟糕地确定优先级。"

建议：
> "我看过团队在一个季度内发布 14 个功能，但没有移动任何关键指标。他们优先考虑活动而非进展。"

### 添加数据
第三个错误将受益于证据：
> "[添加引用]：研究表明，没有定期用户联系的团队构建未使用功能的可能性高 3 倍 [需要来源]"

### 流程改进
考虑重新排序：错误 3 → 错误 2 → 错误 1
这从小影响构建到大影响。

准备好下一部分！
```

## 写作工作流程

### 博客文章工作流程
1. 一起制定大纲
2. 研究关键点
3. 撰写介绍 → 获取反馈
4. 撰写正文部分 → 每个部分反馈
5. 撰写结论 → 最终审查
6. 润色和编辑

### 通讯工作流程
1. 讨论钩子想法
2. 快速大纲（较短格式）
3. 一次会话起草
4. 审查清晰度和链接
5. 快速润色

### 技术教程工作流程
1. 概述步骤
2. 编写代码示例
3. 添加解释
4. 测试说明
5. 添加故障排除部分
6. 最终审查准确性

### 思想领导力工作流程
1. 头脑风暴独特角度
2. 研究现有观点
3. 发展你的论点
4. 以强烈的观点写作
5. 添加支持证据
6. 精心设计引人注目的结论

## 专业提示

1. **在 VS Code 中工作**：比网络 Claude 更适合长篇写作
2. **一次一个部分**：递增获取反馈
3. **单独保存研究**：保持 research.md 文件
4. **版本化你的草稿**：article-v1.md, article-v2.md 等
5. **大声朗读**：使用反馈识别笨拙的句子
6. **设置截止日期**："我想今天完成草稿"
7. **休息**：写作，获取反馈，暂停，修改

## 文件组织

写作项目的推荐结构：

```
~/writing/article-name/
├── outline.md          # 你的大纲
├── research.md         # 所有研究和引用
├── draft-v1.md         # 第一稿
├── draft-v2.md         # 修订稿
├── final.md            # 可发布版本
├── feedback.md         # 收集的反馈
└── sources/            # 参考材料
    ├── study1.pdf
    └── article2.md
```

## 最佳实践

### 研究
- 引用前验证来源
- 尽可能使用最近的数据
- 平衡不同的观点
- 链接到原始来源

### 反馈
- 具体说明你想要什么："这技术含量太高吗？"
- 分享你的担忧："我担心这部分拖沓"
- 提问："这逻辑流程吗？"
- 请求替代方案："解释这的另一种方式是什么？"

### 声音
- 分享你的写作例子
- 指定语调偏好
- 指出好的匹配："那听起来像我！"
- 标记不匹配："对我的风格来说太正式"
