# CLI 创建器

创建一个真正的 CLI，未来的 Codex 线程可以通过命令名从任何工作目录运行。

此技能用于持久工具，而非一次性脚本。如果当前仓库中的短脚本可以解决任务，请在那里编写脚本。

## 开始

为目标工具、其源代码和它应该执行的第一个实际任务命名：

- 源代码：API 文档、OpenAPI JSON、SDK 文档、curl 示例、浏览器应用、现有内部脚本、文章或工作的 shell 历史记录。
- 任务：字面意义的读写操作，如 `list drafts`、`download failed job logs`、`search messages`、`upload media`、`read queue schedule`。
- 安装名称：简短的二进制名称，如 `ci-logs`、`slack-cli`、`sentry-cli` 或 `buildkite-logs`。

当用户想要个人工具且未命名仓库时，首选在 `~/code/clis/<tool-name>` 下创建新文件夹。

在搭建脚手架之前，检查提议的命令是否已存在：

```bash
command -v <tool-name> || true
```

如果存在，选择更清晰的安装名称或询问用户。

## 选择运行时

选择之前，检查用户的机器和源材料：

```bash
command -v cargo rustc node pnpm npm python3 uv || true
```

然后选择最不令人惊讶的工具链：

- 对于 Codex 应该从任何仓库运行的持久 CLI，默认使用 **Rust**：一个快速的二进制文件，强大的参数解析，良好的 JSON 处理，易于复制/安装到 `~/.local/bin`。
- 当官方 SDK、认证助手、浏览器自动化库或现有仓库工具是 CLI 可以更好的原因时，使用 **TypeScript/Node**。
- 当源代码是数据科学、本地文件转换、笔记本、SQLite/CSV/JSON 分析或可以作为持久命令安装的 Python 密集型管理工具时，使用 **Python**。

除非能实质性地改善 CLI，否则不要选择会增加设置摩擦的语言。如果最佳语言未安装，要么在用户批准的情况下安装缺失的工具链，要么选择下一个最佳的已安装选项。

在搭建脚手架之前，用一句话说明选择，包括原因和您找到的已安装工具链。

## 命令约定

在编码之前，在聊天中勾勒命令界面。包括二进制名称、发现命令、解析或 ID 查找命令、读取命令、写入命令、原始转义舱口、认证/配置选择以及 PATH/安装命令。

设计命令界面时，请阅读 [references/agent-cli-patterns.md](references/agent-cli-patterns.md) 了解预期的可组合 CLI 形状。

构建朝向此界面：

- `tool-name --help` 显示每个主要功能。
- `tool-name --json doctor` 验证配置、认证、版本、端点可达性和缺失的设置。
- `tool-name init ...` 在仅环境认证很痛苦时存储本地配置。
- 发现命令查找账户、项目、工作区、团队、队列、频道、仓库、仪表板或其他顶级容器。
- 解析命令将名称、URL、slug、永久链接、客户输入或构建链接转换为稳定 ID，以便未来命令不会重复广泛搜索。
- 读取命令获取精确对象并列出/搜索集合。分页列表支持有限的 `--limit`、游标、偏移量或明确记录的默认值。
- 写入命令每个执行一个命名操作：创建、更新、删除、上传、调度、重试、评论、草稿。它们接受最窄的稳定资源 ID，在服务允许时首先支持 `--dry-run`、`draft` 或 `preview`，并且不会在 `fix`、`debug` 或 `auto` 等广泛命令中隐藏写入操作。
- `--json` 返回稳定的机器可读输出。
- 存在原始转义舱口：`request`、`tool-call`、`api` 或最接近的诚实名称。

不要只暴露通用的 `request` 命令。为重复的任务提供高级动词。

在 CLI README 或等效文件中记录 JSON 策略：API 传递与 CLI 封装、成功形状、错误形状以及每个命令系列的一个示例。在 `--json` 下，错误必须是机器可读的，且不得包含凭证。

## 认证和配置

首先支持简单的路径，按以下优先顺序：

1. 使用服务标准名称的环境变量，如 `GITHUB_TOKEN`。
2. `~/.<tool-name>/config.toml` 下的用户配置或其他简单的文档化路径。
3. `--api-key` 或工具特定的令牌标志仅用于显式的一次性测试。对于正常使用，首选环境/配置，因为标志可能会泄漏到 shell 历史或进程列表中。

永远不要打印完整的令牌。`doctor --json` 应该说明是否有令牌可用、认证源类别（`flag`、`env`、`config`、提供商默认值或缺失）以及缺失的设置步骤。

如果 CLI 可以在没有网络或认证的情况下运行，请在 `doctor --json` 中明确说明：报告夹具/离线模式、是否找到夹具数据以及该模式是否不需要认证。

对于源自 DevTools curl 的内部 Web 应用，在实现之前创建清理的端点注释：资源名称、方法/路径、必需的标头、认证机制、CSRF 行为、请求体、响应 ID 字段、分页、错误和一个编辑过的示例响应。永远不要提交复制的 cookie、承载令牌、客户密钥或完整的生产负载。

使用屏幕截图推断工作流程、UI 词汇、字段和确认点。除非与网络请求、导出、文档页面或夹具配对，否则不要将屏幕截图视为 API 证据。

## 构建工作流程

1. 读取源代码足够多，以盘点资源、认证、分页、ID、媒体/文件流、速率限制和危险的写入操作。如果文档公开 OpenAPI，请在命名命令之前下载或检查它。
2. 在聊天中勾勒命令列表。保持名称简短且对 shell 友好。
3. 用 README 或等效的仓库面向说明搭建 CLI 脚手架。
4. 实现 `doctor`、发现、解析、读取命令、如果请求的话一个狭窄的草稿或干运行写入路径，以及原始转义舱口。
5. 在 PATH 上安装 CLI，以便 `tool-name ...` 在源文件夹外工作。
6. 从另一个仓库或 `/tmp` 进行冒烟测试，而不仅仅是使用 `cargo run` 或包管理器包装器。运行 `command -v <tool-name>`、`<tool-name> --help` 和 `<tool-name> --json doctor`。
7. 运行格式化、类型检查/构建、请求构建器的单元测试、分页/请求体构建器、无认证 `doctor`、帮助输出，以及至少一个夹具、干运行或实时只读 API 调用。

如果需要实时写入以获得信心，请先询问并使其可逆或仅草稿。

当源代码是现有脚本或 shell 历史记录时，将工作调用拆分为实际阶段：设置、发现、下载/导出、转换/索引、草稿、上传、轮询、实时写入。保留用户已经依赖的标志、路径和环境变量，然后用稳定 ID、有界 JSON 和文件输出来包装可重复的阶段。

对于原始转义舱口，首先支持只读调用。除非用户要求特定的写入，否则不要对实时服务运行原始非 GET/HEAD 请求。

对于媒体、工件或预签名上传流，分别测试每个阶段：创建上传、传输字节、轮询/读取处理状态，然后附加或引用生成的 ID。

对于基于夹具的原型，将夹具保存在可预测的项目路径中，并使 CLI 在安装后定位它们。从 `/tmp` 进行冒烟测试，以捕获仅在源文件夹内工作的二进制文件。

对于面向日志的 CLI，将确定性片段提取与模型解释分开。首选一个发出文件名、行号或字节范围、匹配规则和简短摘录的命令。

## Rust 默认值

在 Rust 中构建时，使用已建立的 crate 而不是自定义解析器：

- `clap` 用于命令和帮助
- `reqwest` 用于 HTTP
- `serde` / `serde_json` 用于负载
- `toml` 用于小配置文件
- `anyhow` 用于 CLI 形状的错误上下文

添加 `Makefile` 目标，如 `make install-local`，构建发布版本并将二进制文件安装到 `~/.local/bin`。

## TypeScript/Node 默认值

在 TypeScript/Node 中构建时，保持 CLI 可作为普通命令安装：

- `commander` 或 `cac` 用于命令和帮助
- 原生 `fetch`、官方 SDK 或用户现有的 HTTP 助手用于 API 调用
- `zod` 仅在外部负载验证防止真正破坏的地方
- `package.json` `bin` 条目用于已安装的命令
- `tsup`、`tsx` 或 `tsc` 使用仓库的现有约定

添加安装路径，如 `pnpm install`、`pnpm build` 和 `pnpm link --global`，或一个 `Makefile` 目标，将小包装器安装到 `~/.local/bin`。

## Python 默认值

在 Python 中构建时，除非工作流需要更多，否则首选简单的标准库组件：

- `argparse` 用于命令和帮助，或当子命令否则会变得混乱时使用 `typer`
- `urllib.request` / `urllib.parse`、`requests` 或 `httpx` 用于 HTTP，匹配已经安装或在附近已经使用的内容
- `json`、`csv`、`sqlite3`、`pathlib` 和 `subprocess` 用于本地文件、导出、数据库和现有脚本
- `pyproject.toml` 控制台脚本或小的可执行包装器用于已安装的命令
- `uv` 或虚拟环境仅在实际需要依赖项时

添加 `Makefile` 目标，如 `make install-local`，将命令安装到 PATH 上，并记录它是否依赖于 `uv`、虚拟环境或仅系统 Python。

## 配套技能

CLI 工作后，为其创建或更新一个小技能。当 `$skill-creator` 可用时使用它。对于个人配套技能，使用 `$CODEX_HOME/skills/<tool-name>/SKILL.md`，除非用户命名仓库本地的 `.codex/skills/...` 路径或另一个技能仓库。

按照未来 Codex 线程应该使用 CLI 的顺序编写配套技能，而不是作为每个功能的导览。解释：

- 如何验证已安装的命令存在。
- 首先应该运行哪个命令。
- 如何配置认证。
- 哪个发现命令找到公共 ID。
- 安全读取路径。
- 预期的草稿/写入路径。
- 原始转义舱口。
- 没有明确用户批准不应该做什么。
- 三个可复制粘贴的命令示例。

将 API 参考详细信息保存在 CLI 文档或技能参考文件中。保持技能专注于排序、安全性和未来 Codex 线程实际应该运行的示例。
